Summary of investigation so far
------------------------------------------------------------------
Key technical facts from the debug log (cloak_debug.log, attached):

  1. Date stamp is 20260613 (UTC) — machine clock is correct, so this
     is not a date-skew issue.
  2. Keys load with the expected lengths (public 20 chars, private 40
     chars) — not a .env parse problem.
  3. Canonical request includes the content-type header only, lowercase,
     matching the documented CLOAK-AUTH scheme.
  4. Server is reachable and returns the Access-Control-Allow-Origin
     header. Endpoint URL is alive and the request body is well-formed
     JSON, so the failure is specifically in the signature / credential
     check.

This narrows the likely cause to one of three things, which is what the
questions below are trying to confirm:

  - The documented CLOAK-AUTH signing scheme has been updated since my
    reference implementation was written.
  - The SignedHeaders set needs to include more headers (e.g., accept,
    x-cloak-service, host) and not just content-type.
  - An account-side activation step is required beyond key issuance.
